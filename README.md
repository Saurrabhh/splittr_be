# Splittr Backend (`splittr_be`)

A Go-based backend service for **Splittr**, a bill-splitting application. This service features clean architecture, type-safe SQL interactions using `sqlc`, schema migrations with `goose`, live reloading via `air`, and token-based authentication integrated with the Firebase Admin SDK.

---

## 🚀 Tech Stack

- **Language:** Go (1.26.3)
- **Database:** PostgreSQL (16)
- **Routing:** [go-chi/chi](https://github.com/go-chi/chi) (v5)
- **SQL Generator:** [sqlc](https://sqlc.dev/) (v2)
- **Migrations:** [goose](https://github.com/pressly/goose)
- **Authentication:** Firebase Admin SDK
- **Hot Reload:** [air](https://github.com/air-verse/air)

---

## 📂 Project Structure

```text
├── cmd/
│   └── api/                  # Application entry point, router, and dependencies injection
├── internal/
│   ├── auth/                 # Firebase authentication middleware and token verifiers
│   ├── config/               # Application configuration parsing (.env files and env vars)
│   ├── db/                   # Database connection pool, transaction manager, migrations, and queries
│   │   ├── dbgen/            # Go code generated by sqlc from raw SQL
│   │   ├── migrations/       # goose SQL schema migration files
│   │   └── queries/          # Raw SQL queries compiled by sqlc
│   ├── response/             # Standardized API JSON error and success responses
│   └── user/                 # User domain business logic (handler, usecase, repository)
├── .air.toml                 # Air tool configuration for local hot reload
├── docker-compose.yml        # PostgreSQL service container definition
├── go.mod                    # Go dependencies
├── sqlc.yaml                 # Configuration for sqlc code generation
└── .env.example              # Example environment configurations
```

## 🛠️ Configuration & Setup

Follow these steps to configure and spin up the project locally:

### 1. Clone & Set Up Environment Variables
Copy the `.env.example` file to create your own local configuration file:
```bash
cp .env.example .env
```

The `.env` file contains:
- `PORT`: Port on which the API server will run (defaults to `8080`).
- `DATABASE_URL`: Connection URI for PostgreSQL.
- `GOOGLE_APPLICATION_CREDENTIALS`: Path to your Firebase service account private key file.

### 2. Configure Firebase Admin SDK
1. Go to your **Firebase Console** -> **Project Settings** -> **Service Accounts**.
2. Generate a new private key and download the JSON file.
3. Save the downloaded file to the root of the project with the filename `firebase-key.json` (this file is already included in `.gitignore` to prevent leaking credentials).

### 3. Spin up PostgreSQL
Use Docker Compose to run a local PostgreSQL instance in the background:
```bash
docker compose up -d
```

### 4. Run Schema Migrations
Apply database migrations using `goose` against your PostgreSQL database.

Since `goose` natively supports configuration via environment variables, we define `GOOSE_DRIVER`, `GOOSE_DBSTRING`, and `GOOSE_MIGRATION_DIR` in the `.env` file. You can load these variables into your shell session and run `goose up` directly.

# Run migrations
```
goose up
```

*(You can run other goose commands similarly: `goose status`, `goose down`, etc. without having to pass connection parameters.)*

### 5. Generate Database Code (Optional)
If you add or update SQL queries inside `internal/db/queries/` or migration schemas in `internal/db/migrations/`, generate the Go boilerplate using `sqlc`:
```bash
sqlc generate
```

## 🏃 Running the Application

### Development (with Hot Reload)
Run the application with `air`. It will automatically watch for file changes, rebuild the binary on-the-fly, and restart the API server:
```bash
air
```

### Production Build & Run
To compile a production binary and execute it:
```bash
# Build binary
go build -o bin/api ./cmd/api

# Run binary
./bin/api
```

## 🔌 API Endpoints

All domain endpoints (except `/health`) require authenticating with a **Firebase ID Token** sent via the `Authorization` header as a Bearer token:

```text
Authorization: Bearer <FIREBASE_ID_TOKEN>
```

| Method | Endpoint | Description | Auth Required | Request Body |
| :--- | :--- | :--- | :---: | :--- |
| **GET** | `/health` | Server health status | No | N/A |
| **POST** | `/users` | Registers/creates a new user profile | Yes | `{"name": "John Doe"}` |
| **GET** | `/users/me` | Retrieves the profile of the current user | Yes | N/A |

### Example Request (Registration):
```bash
curl -X POST http://localhost:8080/users \
  -H "Authorization: Bearer YOUR_FIREBASE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name": "Saurabh"}'
```
